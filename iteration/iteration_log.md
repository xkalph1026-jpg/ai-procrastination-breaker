# AI拖延症破解器 — 迭代升级日志

## 迭代 #1: v1.0 → v1.1 （模拟使用3天）

**迭代日期：** 2026-07-16
**使用天数：** 模拟3天（每天3次对话 × 3天 = 9次真实场景）

---

### 发现的问题

#### 问题1：关键词匹配过于死板 ⭐⭐⭐
**现象：** 用户说「怕写不好」→ 分类器判定为空，因为关键词列表里只有「怕做不好」没有「写不好」。
**频率：** 9次中出现了3次分类失败（33%）。
**影响：** 用户第一次使用时分类失败 → 体验极差 → 可能直接放弃。
**根因：** 简单子串匹配，缺少口语变体覆盖。中文表达灵活，"写不好/做不好/怕不行/怕翻车"都是同一类情绪。

#### 问题2：缺少「否定型反馈」处理 ⭐⭐⭐
**现象：** 用户说「我试了你说的方法但是没用」→ AI不知道如何回应，要么重复建议，要么冷场。
**频率：** 9次中出现1次。
**影响：** 用户感到挫败时得不到有效帮助，信任崩塌。
**根因：** Phase 4 中虽然提到了否定反馈，但脚本层面没有对应的处理逻辑，微行动没有「降级」机制。

#### 问题3：微行动模板缺少动态变量填充 ⭐⭐
**现象：** 微行动说「打开文档」而不是「打开论文文档」→ 用户仍需自己翻译成具体动作。
**频率：** 每次使用都出现。
**影响：** 增加用户认知负担——本应「拿来就做」，结果还要自己脑补。这违背了微行动的零摩擦原则。
**根因：** 模板是静态文本，没有嵌入用户实际任务名。

#### 问题4：连胜追踪缺失 ⭐⭐
**现象：** 用户连续3天都用Skill破解了拖延，但没有任何反馈告知这个成就。
**频率：** 跨所有使用。
**影响：** 缺少正向激励循环。用户看不到自己的进步，难以坚持。
**根因：** victory_log 只记录了单次事件，没有计算连续天数。

#### 问题5：第1天初始体验空白 ⭐
**现象：** 新用户第一次用，没有历史模式、没有战果库 → 对话比老用户少了一层「模式识别」的温度。
**频率：** 第1天的3次使用都如此。
**影响：** 新用户体验打折扣，没有「这个工具记得我」的感觉。
**根因：** 模式识别只对老用户生效，新用户的欢迎引导不够。

#### 问题6：多任务并发缺少优先级引导 ⭐
**现象：** 用户说「作业明天截止、论文后天交、pre下周，全都还没做」→ 分类器认为这是「任务过大」，但实际上用户需要的是**优先级排序**，不只是拆分。
**频率：** 9次中出现1次。
**影响：** 分类偏差导致治疗方案不匹配。
**根因：** 「多任务并发」应该被识别为一个独立的拖延子类型或特殊处理分支。

## 迭代 #2: v1.1 → v1.2 （测试驱动 + 继续模拟使用）

**迭代日期：** 2026-07-16
**触发来源：** v1.1 测试记录暴露的6个BUG + 继续使用发现的2个新痛点

---

### 发现的问题

#### 问题1：时间拖延信号词缺失 ⭐⭐⭐
**现象：** "拖了三天"、"拖了一周"、"又两天没动了" → 全部无法触发分类。
**频率：** 15条测试中2条因此失败（TC-01, TC-07）。
**影响：** 用户最自然的表达方式——「拖了N天」——反而无法被理解。
**根因：** 所有类型的关键词只关注「为什么不做的心理原因」，忽略了「拖延行为本身」的信号词。这是设计盲区。

#### 问题2：关键词粒度过细（"烦" vs "好烦"、"分析什么" vs "怎么分析"）⭐⭐⭐
**现象：** "烦"单字不匹配（只有"好烦"）、"分析什么"不匹配（只有"怎么分析"）。
**频率：** 15条测试中2条因此失败（TC-02, TC-10）。
**影响：** 中文表达的灵活多变导致大量漏网。精确子串匹配对中文不友好。
**根因：** 扩展关键词本质仍是精确子串，缺少语义层面的匹配。

#### 问题3："卡住了"未被任何类型覆盖 ⭐⭐
**现象：** "第三章又卡住了，不想写了" → 无匹配。"卡住了"是拖延最直白的表达之一。
**频率：** 出现即失败（TC-07）。
**根因：** "卡住了"是通用卡点信号，应该被所有类型共享。目前每个类型独立管理关键词，缺少共享信号词层。

#### 问题4：正则提取过广 ⭐
**现象：** `写论文，拖了三天了` 被整体捕获为任务名 → 微行动变成「打开写论文，拖了三天了的第一步」。
**根因：** 正则 `写(.{1,8}论文)` 贪婪匹配，没有在标点处截断。

#### 问题5：置信度数值误导 ⭐
**现象：** 置信度显示 0.06、0.07、0.13——但实际上分类是正确的（只是分母太大）。
**影响：** 用户（或调试者）看到0.06会误以为AI"很不确定"，实则关键词已命中。
**根因：** 分母是扩展关键词总数（12-17），命中1个就是 1/15≈0.06。应该用命中数/核心词数作为置信度。

#### 问题6：成功型跟进消息无法处理 ⭐
**现象：** "按你说的写了5分钟，感觉还不错，接下来做什么" → 无分类，无回应。
**频率：** 使用2天后出现。
**影响：** 用户做完微行动后处于高动力状态，此时最需要趁热打铁——但Skill沉默了。
**根因：** 缺少「跟进状态」检测。应该识别成功汇报 → 提出下一个微行动。

#### 问题7：微行动文本偶有语法不通 ⭐
**现象：** "打开做项目，一周了还没的第一步" —— 提取了太多无关文字。
**根因：** 与问题4同源：正则提取没有做后处理清理。

#### 问题8：仪表盘在没有失败记录时仍显示成功率 ⭐
**现象：** 成功率为空 → 仪表盘仍尝试显示。
**根因：** 边界条件处理不完整。

---

### 迭代方向

| # | 改进项 | 解决哪些BUG | 优先级 |
|---|--------|-----------|--------|
| 1 | **共享信号词层** — 所有类型共享的通用拖延信号 | BUG-01, BUG-03 | P0 |
| 2 | **模糊匹配升级** — 字符级包含匹配（"烦" in "好烦"） | BUG-02, BUG-04 | P0 |
| 3 | **正则提取优化** — 非贪婪 + 标点截断 + 后处理 | BUG-05, BUG-07 | P1 |
| 4 | **置信度重算** — 命中数/核心词数（非扩展总数） | BUG-05 | P1 |
| 5 | **成功型跟进** — 识别汇报成功的消息，提议下一个微行动 | NEW-06 | P1 |
| 6 | **仪表盘边界修复** — 无失败记录时的显示处理 | BUG-06 | P2 |
| 7 | **反拖延证据库展示** — 用户可查看自己的战果历史 | NEW-07 | P2 |
| 8 | **每日回顾提示** — 当天第一次使用时告知昨日战绩 | NEW-08 | P2 |

---

### 改进实施

#### 改动1：共享信号词层（P0）✅
新增 `SIGNAL_WORDS` 字典，独立于类型关键词：

```python
SIGNAL_WORDS = {
    "temporal": ["拖了", "又拖", "一直拖", "还没开始", "好几天了", "过了N天", "又没做"],
    "stuck": ["卡住了", "卡着", "卡壳", "卡在", "停滞", "停住了", "做不下去"],
    "avoidance": ["不想写", "不想做", "不想动", "不想干", "不想开始"],
}
```

这些信号词不直接决定类型，但能**触发分类流程**（解决"无匹配"问题），然后结合追问确定类型。

#### 改动2：模糊匹配升级（P0）✅
- "烦" → 匹配 `any("烦" in kw for kw in expanded_keywords)` ✅
- "分析什么" → 匹配"怎么分析"（字符重叠度≥60%触发） ✅
- 短输入（≤2字）→ 特殊处理，直接触发追问而非分类

#### 改动3：正则提取优化（P1）✅
```python
# v1.1 (贪婪): 写(.{1,8}论文) → 捕获"写论文，拖了三天了"
# v1.2 (非贪婪+截断): 写(.{1,8}?论文) + 按标点截断 → "写论文"
```

#### 改动4：置信度重算（P1）✅
```python
# v1.1: confidence = len(matched) / len(all_expanded_keywords)  # → 0.06
# v1.2: confidence = len(matched) / len(core_keywords)          # → 0.2+
```

#### 改动5：成功型跟进（P1）✅
新增 `is_success_report()` 检测：
- "做完了"、"写完了"、"还不错"、"接下来"、"继续"
- 触发后 → 共情庆祝 → 记录战果 → 提议下一步

#### 改动6-8：仪表盘 + 证据库 + 每日回顾（P2）✅
- 仪表盘成功率边界修复
- `--history` 命令展示反拖延证据库
- 每日首次使用时显示昨日总结
